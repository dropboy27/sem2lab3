# Лабораторная работа 2
## Модель задачи: дескрипторы и @property

* **Цель:** Освоить управление доступом к атрибутам и защиту инвариантов доменной модели с помощью пользовательских дескрипторов и свойств (`property`).

* **Библиотеки:** `pytest`, `typing`, `datetime`, `random`, `time`, `unittest.mock`

* **Допущения:**
  - Идентификатор задачи (`id`) — целое положительное число, не изменяется после создания.
  - Описание (`description`) — непустая строка, не изменяется после создания.
  - Приоритет (`priority`) — целое число от 1 до 5, не изменяется после создания.
  - Статус (`status`) — одно из значений: `'новая'`, `'в работе'`, `'завершена'`. Может изменяться.
  - Время создания (`created_at`) — автоматически устанавливается при создании и доступно только для чтения.
  - Вычисляемое свойство `is_active` — `True`, если статус не равен `'завершена'`.
  - Краткое описание (`short_desc`) — первые 10 символов описания (реализовано через non‑data дескриптор, допускает переопределение на уровне экземпляра).
  - Некорректные состояния объекта невозможны благодаря валидации в дескрипторах и генерации исключения `TaskError`.

* **Чему я научился:**
  - Создание **data‑дескрипторов** (`ReadOnlyInt`, `ReadOnlyString`, `PriorityField`, `StatusField`) для инкапсуляции логики валидации и управления доступом.
  - Использование **non‑data дескриптора** (`ShortDescription`) для предоставления вычисляемого значения с возможностью переопределения в экземпляре.
  - Применение `@property` для защищённых (`created_at`) и вычисляемых (`is_active`) атрибутов.
  - Разделение публичного API и внутреннего состояния (приватные атрибуты `_id`, `_description`, и т.д.).
  - Генерация специализированных исключений (`TaskError`) при нарушении инвариантов.
  - Реализация протокола `TaskSource` для унифицированного получения задач из разных источников (файл, генератор, API‑заглушка).
  - Написание модульных тестов с `pytest` для проверки корректности дескрипторов, свойств и обработки ошибок.

### Алгоритм решения
1. Создан класс `Task` с дескрипторами для каждого атрибута, требующего контроля:
   - `ReadOnlyInt` – проверяет тип, положительность и запрещает изменение.
   - `ReadOnlyString` – проверяет тип, непустоту и запрещает изменение.
   - `PriorityField` – проверяет диапазон и запрещает изменение.
   - `StatusField` – разрешает изменение, но только среди допустимых значений.
   - `ShortDescription` – non‑data дескриптор, возвращает первые 10 символов описания (или всё описание, если оно короче).
2. В конструкторе `__init__` выполняется начальная установка значений через дескрипторы, что автоматически запускает валидацию.
3. Добавлены свойства:
   - `created_at` – только для чтения, хранит момент создания.
   - `is_active` – вычисляется на основе текущего статуса.
4. Все исключения при неверных данных объединены в иерархию `TaskError`.
5. Для единообразного получения задач определён протокол `TaskSource` (использован `typing.Protocol` с `runtime_checkable`).
6. Реализованы три источника:
   - `FileTaskSource` – читает задачи из текстового файла.
   - `TaskGenerator` – генерирует случайные задачи.
   - `ApiTaskSource` – имитирует задержку и возвращает задачи.
7. Функция `receive` принимает любой объект, реализующий `TaskSource`, и возвращает список задач.
8. В `main.py` реализовано консольное меню для выбора источника и просмотра задач.

### Интересные тест-кейсы
- Попытка создать задачу с отрицательным `id` – должно возникнуть `TaskError`.
- Попытка изменить `id` после создания – `TaskError`.
- Установка статуса, не входящего в допустимый набор – `TaskError`.
- Присвоение приоритета вне диапазона 1–5 – `TaskError`.
- Пустое описание – `TaskError`.
- Проверка свойства `is_active` при изменении статуса.
- Поведение non‑data дескриптора `short_desc`: после присваивания экземпляру возвращается новое значение, после удаления – снова первые 10 символов описания.
- Чтение из несуществующего файла – ожидаемое исключение `FileTaskSourceNotFound`.
- Генерация задач через `TaskGenerator` – все поля соответствуют допустимым диапазонам.
- Имитация задержки в `ApiTaskSource` – проверка, что `time.sleep` вызывается.

### Инструкция к использованию

#### Запуск приложения
```bash
python -m src.main
```
Меню

- Прочитать задачу из файла – запрашивает имя файла, ожидает строки формата:
<id> <описание> <приоритет> <статус>
(например: 1 Do_homework 3 новая). Некорректные строки пропускаются с выводом ошибки.

- Сгенерировать задачи – запрашивает количество, создаёт задачи со случайными данными.

- API заглушка – запрашивает количество, имитирует сетевую задержку и возвращает задачи.

- Увидеть задачи – выводит все накопленные задачи в полном или кратком формате.

- Выход – завершение программы.
